# MySQL Relationship Mapping & Foreign Key Rules

In MySQL, designing database relationships is all about how tables talk to one another. Think of it as defining the social rules for your data.

To connect two tables, we use a **Foreign Key (FK)**, which is a column in one table that points to the **Primary Key (PK)** of another table.

---

## 1. Relationship Mapping

### One-to-One (1:1)

In a 1:1 relationship, a record in Table A can connect to _only one_ record in Table B, and vice versa.

- **How to map it:** Put the Foreign Key in either table, but make that Foreign Key column **UNIQUE**.
- **Example:** A `User` has exactly one `User_Profile`.

```sql
CREATE TABLE users (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50)
);

CREATE TABLE user_profiles (
    profile_id INT PRIMARY KEY AUTO_INCREMENT,
    bio TEXT,
    user_id INT UNIQUE, -- UNIQUE guarantees the 1:1 relationship
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);
```

### One-to-Many (1:M)

This is the most common relationship. A record in Table A can connect to _multiple_ records in Table B, but a record in Table B connects to _only one_ in Table A.

- **How to map it:** Always put the Foreign Key on the **"Many" side** of the relationship.
- **Example:** One `Department` has many `Employees`. (The Employee table gets the `department_id`).

```sql
CREATE TABLE departments (
    dept_id INT PRIMARY KEY AUTO_INCREMENT,
    dept_name VARCHAR(50)
);

CREATE TABLE employees (
    emp_id INT PRIMARY KEY AUTO_INCREMENT,
    emp_name VARCHAR(50),
    dept_id INT, -- No UNIQUE constraint here, because many employees can share a dept
    FOREIGN KEY (dept_id) REFERENCES departments(dept_id)
);
```

---

## 2. Foreign Key Rules (Referential Integrity)

What happens if you delete a department that still has employees? Or change a user's ID? MySQL needs rules to handle these situations. You define these using `ON DELETE` and `ON UPDATE` clauses.

| Rule            | What happens to the Child Table (the "Many" side)?                                                        | Best Used For...                                                                                             |
| :-------------- | :-------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------- |
| **`CASCADE`**   | If the parent row is deleted/updated, the child rows are **automatically deleted/updated** along with it. | Dependent data (e.g., deleting a blog post deletes all its comments).                                        |
| **`SET NULL`**  | If the parent row is deleted/updated, the child rows set their foreign key column to **`NULL`**.          | Optional relationships (e.g., deleting a department sets the employee's department to "None").               |
| **`RESTRICT`**  | MySQL **blocks** the delete/update on the parent table if any child data relies on it.                    | Strict data protection (e.g., you can't delete a product if it's currently inside an active customer order). |
| **`NO ACTION`** | In MySQL, this does the exact same thing as `RESTRICT`. It **blocks** the action.                         | Standard default behavior.                                                                                   |

### How to Write It in SQL

Here is how you actually apply these rules when creating a table:

```sql
CREATE TABLE comments (
    comment_id INT PRIMARY KEY AUTO_INCREMENT,
    comment_text TEXT,
    post_id INT,
    -- If the post is deleted, automatically delete this comment (CASCADE)
    FOREIGN KEY (post_id) REFERENCES posts(post_id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);
```
